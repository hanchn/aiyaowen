# 架构重构计划：Feature-First (特性优先) 架构

为了解决“目录命名丑陋”和“单页面聚合”的问题，我建议采用 **Routing vs. Implementation Separation (路由与实现分离)** 的模式。

## 核心理念
1.  **`app/` 目录 (路由层)**：只负责定义 URL 和页面跳转。保持极致的扁平、简洁。不要在这里写复杂的业务代码。
2.  **`src/features/` 目录 (业务层)**：真正的代码所在地。每个页面/功能模块一个文件夹，实现“单页面聚合”。

## 目标结构
```text
/
├── app/                  # 路由层 (只留入口文件)
│   ├── (tabs)/
│   │   ├── index.tsx     # 引用 @/features/home
│   │   └── explore.tsx   # 引用 @/features/explore
│   ├── modal.tsx         # 引用 @/features/modal
│   └── _layout.tsx
│
├── components/           # 全局通用组件 (多页面分离的公共部分)
│   ├── ui/
│   └── ...
│
└── features/             # 核心业务层 (单页面聚合)
    ├── home/             # 首页的所有逻辑聚合在这里
    │   ├── components/   # 首页独有的组件
    │   ├── hooks/        # 首页独有的 hooks
    │   └── index.tsx     # 首页主视图
    ├── explore/
    │   └── ...
    └── modal/
        └── ...
```

## 执行步骤
1.  **创建 `features` 目录**：建立 `features/home`, `features/explore`, `features/modal`。
2.  **迁移代码**：将 `app/(views)` 下的业务逻辑移动到对应的 `features` 目录中。
3.  **重置 `app` 目录**：删除冗余的 `(views)` 文件夹，恢复 `app` 的扁平结构。
4.  **连接路由**：修改 `app` 下的文件，让它们作为“代理”直接引用 `features` 中的组件。

这种方案既利用了 Expo Router 的路由能力，又避开了它的命名限制，让你的业务代码目录清爽、可读性强。是否立即开始重构？
